## 2. Current Testing Priorities

### 2.1 Reliability and fresh-audit confidence

**Status**: **ACTIVE**  
**Priority**: High

**Goal**: Make the full audit and normal test runs trustworthy enough that a failure usually means a real regression, not test harness noise.

**Current work**:
- [ ] Run a fresh full audit after each significant test-harness change:
  - `python development_tools/run_development_tools.py audit --full`
- [ ] When a failure appears flaky, capture:
  - failing node
  - command used
  - whether it failed serially, parallel, or both
  - most recent recurrence date
  - current hypothesis
- [ ] Keep active suspect lists short. Close stale suspects when they have not recurred in recent audit/test evidence.

**Acceptance**:
- [ ] Full Tier 3/audit path is clean on consecutive relevant reruns.
- [ ] Any recurring failure has repro metadata and an owner/hypothesis.

---

### 2.2 Harness isolation and log cleanliness

**Status**: **MONITORING**  
**Priority**: High

**Goal**: Tests should not pollute production logs, leave uncontrolled artifacts, or depend on shared mutable paths.

**Completed baseline**:
- Test/dev-tools logging was routed to test/dev-tools log locations.
- Windows temp/ACL hardening was added to reduce `PermissionError` / `Access is denied` cleanup failures.
- Retention/archive path handling was hardened to avoid repeated long-name backup/archive failures.
- `logs/ai_dev_tools.log` may intentionally mirror dev-tools audit entries for operator visibility.

**Current work**:
- [ ] Confirm fresh runs do not write normal test traffic to production logs except the intentional dev-tools mirror.
- [ ] Monitor fresh test logs for recurring `Access is denied`, `PermissionError`, or `WinError 123` failures.
- [ ] Continue migrating tests that write shared paths to per-test `tmp_path` or fixture-owned directories.
- [ ] Investigate any `xdist_worker_crash_output` recurrence before widening parallel execution.

**Acceptance**:
- [ ] Fresh full-audit/test-tool runs show no new production-log pollution outside the intentional mirror.
- [ ] Fresh full-audit/test-tool runs show no recurring Windows cleanup/path errors.
- [ ] New tests use isolated file/log paths by default.

### 2.3 Domain-attribution pytest markers

**Status**: **ACTIVE**  
**Priority**: High/Medium

**Goal**: Add domain markers so audit tooling can tie tests to product areas without relying only on file-path inference.

**Why it matters**: Domain markers support selective execution, coverage invalidation, and more useful AI-generated priorities.

**Current implementation context**:
- `development_tools/tests/domain_mapper.py` and the development-tools config define the domain mapping/taxonomy.
- `development_tools/tests/analyze_test_markers.py` reports marker gaps.
- `AI_PRIORITIES` can surface missing domain-attribution markers when analysis finds gaps.

**Work**:
- [ ] Run the marker analysis before editing:
  - `python development_tools/tests/analyze_test_markers.py --check`
- [ ] Add at least one domain-attribution marker per relevant test/class/file, aligned with the domain mapper.
- [ ] Keep category markers separate from domain markers:
  - category markers describe suite layer, such as `unit`, `behavior`, `integration`, `ui`
  - domain markers describe product area, such as communication/core/ui/tasks/notebook/etc.
- [ ] Roll out in small waves by directory or product domain.
- [ ] Update/verify docs after marker policy becomes stable enough to enforce.

**Acceptance**:
- [ ] Marker analysis reports fewer missing domain-attribution markers after each wave.
- [ ] No category-marker fixes are confused with domain-marker fixes.
- [ ] Policy remains warn-first until the missing-marker count is low enough for enforcement.

### 2.4 Policy boundary tests

**Status**: **ACTIVE**  
**Priority**: Medium

**Goal**: Add static policy tests that prevent known architectural drift from returning.

#### Time API boundary

- [ ] Add policy tests that flag direct `datetime.now()`, `strftime()`, and `strptime()` usage outside `core/time_utilities.py`, except for explicit documented exceptions.
- [ ] Include tests for source scanning behavior and allowlist behavior.

**Why it matters**: Prevents timestamp drift, inconsistent persisted formats, and non-deterministic tests.

#### User-data JSON access boundary

- [ ] Add policy tests that flag ad-hoc per-user `json.load()` / `json.dump()` access outside approved storage/user-data helpers.
- [ ] Use the current storage package names in the allowlist, including approved modules such as:
  - `storage/user_data_read.py`
  - `storage/user_data_write.py`
  - `storage/user_data_registry.py`
  - `storage/user_item_storage.py`
  - other low-level storage modules only if they are deliberately approved
- [ ] Avoid old references to `core/user_data_*` unless compatibility wrappers actually exist and are still approved.

**Why it matters**: Prevents bypassing validation, normalization, backup, cache, and persistence invariants.

#### Core/UI import boundary

- [ ] Add a static policy test that fails if `core/**/*.py` imports `ui.*`.
- [ ] Allow no exceptions unless the exception is temporary, documented, and has a removal plan.

**Why it matters**: Keeps core business logic independent from the PySide/admin UI layer.

**Acceptance**:
- [ ] Each policy has a focused test file.
- [ ] Allowlists are small, explicit, and documented.
- [ ] Failure messages tell the developer exactly what to change.

### 2.5 No-parallel marker reduction

**Status**: **PLANNED / OPPORTUNISTIC**  
**Priority**: Medium

**Goal**: Reduce `@pytest.mark.no_parallel` only when doing so does not create flakiness.

**Current rule**: Stability beats speed. Keep the marker if removing it risks false failures.

**Before any wave**:
- [ ] Refresh the current count:
  - `rg '@pytest.mark.no_parallel' tests/ -n`
- [ ] Identify highest-value candidates from current audit/test behavior, not from stale February lists.
- [ ] For each candidate file:
  - use unique per-test identifiers
  - isolate file writes into fixture-owned paths
  - patch shared module state/constants explicitly
  - run repeated `pytest -n auto` validation before removing the marker
  - add a reason comment if the marker remains

**Acceptance**:
- [ ] Marker count decreases only after stability evidence.
- [ ] Remaining markers have understandable reasons when practical.

### 2.6 Coverage stability and growth

**Status**: **ACTIVE / ONGOING**  
**Priority**: Medium

**Goal**: Grow coverage in important product areas while avoiding coverage work that creates fragile tests.

**Current guidance**:
- Use `development_docs/TEST_COVERAGE_REPORT.md` and `AI_PRIORITIES.md` for live targets.
- Prefer coverage in currently active/high-value areas:
  - communication and Discord flows
  - check-in flow
  - tasks
  - notebook
  - core storage/persistence boundaries
  - UI only where tests are stable and worthwhile
- Deprioritize AI/context coverage until the AI overhaul makes those surfaces more stable.
- Development-tools coverage is tracked primarily in the AI development tools roadmap, not duplicated here.

**Work**:
- [ ] Pick small, behavior-relevant coverage slices from live coverage reports.
- [ ] Avoid percentage-only tests that lock in bad design or brittle implementation details.
- [ ] Track coverage variance only when it blocks confidence in the reports.

**Acceptance**:
- [ ] Coverage trend improves without new flakiness.
- [ ] New tests validate useful behavior, not just lines.

### 2.8 Intermittent failure triage

**Status**: **MONITORING**  
**Priority**: Low/Medium

**Goal**: Keep a useful suspect list without letting old one-off failures become permanent noise.

**Current rule**: Do not act on the old February suspect list without refreshing it against current logs/audit output.

**Work**:
- [ ] Rebuild the active suspect queue from current evidence when a failure recurs.
- [ ] For each open suspect, record:
  - test node
  - last observation date
  - repro command
  - serial/parallel behavior

- [ ] Close stale suspects with a date and evidence note.

**Known recurring/special cases to keep in mind**:

**Acceptance**:
- [ ] Active suspect queue contains only recent, reproducible, or high-signal items.
- [ ] Monitor-only items do not distract from current failures.

## 4. Validation Gates

Use these gates depending on the change type.

### Full reliability gate

```powershell
python development_tools/run_development_tools.py audit --full
```

Use after broad test harness, storage, command routing, scheduling, or audit-tooling changes.

### Targeted failing-node gate

```powershell
pytest path\to\test_file.py::TestClass::test_name -n auto
```

For flaky fixes, repeat the target several times in the same execution mode that failed.

### No-parallel conversion gate

```powershell
pytest path\to\test_file.py -n auto
```

Run after isolating shared paths/state and before removing `@pytest.mark.no_parallel`.

### Marker analysis gate

```powershell
python development_tools/tests/analyze_test_markers.py --check
```

Use before and after domain-marker rollout waves.

### Coverage gate

```powershell
python development_tools/run_development_tools.py audit --full
```

Then review:

- `development_docs/TEST_COVERAGE_REPORT.md`
- `development_tools/AI_STATUS.md`
- `development_tools/AI_PRIORITIES.md`

### Logging/isolation gate

After a test-tool or full-audit run:

- inspect `tests/logs/`
- inspect `development_tools/tests/logs/`
- confirm production logs were not polluted except the intentional dev-tools mirror
- confirm generated `tests/data/` artifacts are under expected/allowlisted roots
